<objective>
Wire the Phase 2 per-fact verification gate into the existing offline validation harness (`scripts/validate-data.mjs`), extending the Phase 1 gate chain. The gate imports the pure checks from `lib/verification.mjs`, reads the committed `.cache/citation-verdicts.json` (never the network, Pitfall 3), and calls `deriveVerificationState` per fact to (a) run a status-breakdown build report and (b) drive the internal-consistency assertions. Critically, and per the D-15 post-critique amendment (R-02): a WITHHELD fact is the normal correct outcome and does NOT fail the build. Distinct-lineage, measure-mismatch and value-divergence are per-fact WITHHOLD reasons computed inside `deriveVerificationState`, never entries in the build-failing error array. The build fails ONLY on genuine internal-consistency violations: `contested` without a matching adjudication outcome, a contested fact whose singular value is non-null (R-05), a pass `sourcesChecked` id absent from the fact's `sources`, a malformed `markedWrong`, and a `derivedFrom` pointer that does not resolve to a real registry id (R-16).

Purpose: This is the moment the trust layer becomes a build invariant. A fact that cannot meet its claim standard is withheld, not asserted, on every build (VRFY-04 continuous) - and one bad fact never stops the rest of the corpus publishing.
Output: The extended gate and the end-to-end corpus-gate tests. This plan owns scripts/validate-data.mjs exclusively.
</objective>
