# Pitfalls Research

**Domain:** Evidence-based UK packaged-food transparency database (provenance-heavy, citation-first, historical reformulation archive)
**Researched:** 2026-06-30
**Confidence:** HIGH on licensing/IP (verified against ODbL text and Open Food Facts terms); HIGH on UK legal framework (training-data domain knowledge of Defamation Act 2013 and Trade Marks Act 1994, not re-litigated here); MEDIUM on data-quality and confidence-model failure modes (synthesised from analogous provenance projects, not a single authoritative post-mortem).

## Critical Pitfalls

### Pitfall 1: ODbL share-alike silently contaminates the whole database

**What goes wrong:**
Open Food Facts data is under the **Open Database License (ODbL)** with a share-alike clause. Teams assume "attribution is enough" and bake OFF fields directly into their core records. The trap is the ODbL distinction between two outputs:
- A **Produced Work** (a web page, chart, comparison rendering generated by querying the data) — requires an **attribution notice only**, no share-alike.
- A **Derivative Database** (a new database built on a substantial part of OFF contents) — triggers **share-alike**: if you publicly use it, you must offer that derivative database under ODbL.

Rendering OFF data on product pages is a Produced Work (fine). But the moment Phase 2 ships a **public API, bulk export, or "journalist toolkit"** that serves the underlying OFF-derived database, that distribution is plausibly a Derivative Database and the share-alike obligation attaches — potentially forcing you to release your curated dataset (including your own painstaking editorial work, if commingled) under ODbL. Discovering this after the data model is built is a rewrite.

**Why it happens:**
The Produced Work vs Derivative Database line is subtle and not obvious from "it's open data". The MVP only displays data (safe), so the obligation stays invisible until the API milestone — by which point OFF fields are entangled with proprietary editorial content in the same tables.

**How to avoid:**
- **Provenance-tag every field at source** so OFF-derived values are always separable from your own editorial facts. The trust layer (source + confidence per fact) already requires this — make `source = Open Food Facts` a first-class, queryable attribute, not a free-text note.
- Treat the public MVP as **Produced Works** and add the required ODbL attribution notice: "Contains information from Open Food Facts, which is made available under the Open Database License (ODbL)" with links to the licence and the source product page.
- Decide the API/export strategy **before** the data model is frozen: either (a) keep OFF-derived data architecturally segregated so a future export of *your* data isn't a derivative of OFF, or (b) accept ODbL for the shared dataset and design for it deliberately.
- The same logic applies to the **CC BY-SA 3.0** licence on OFF **product photos** and the **Database Contents License (DBCL)** on individual contents — all three carry attribution, two carry share-alike.

**Warning signs:**
OFF values stored in the same column as manually-curated values with no source discriminator; an attribution notice that lists "various sources" generically; roadmap discussion of an API or data export with no licence column in the schema.

**Phase to address:**
Foundation / data-model phase (provenance tagging + source as first-class field). Re-verify at the Phase 2 API/export milestone before any database is distributed.

### Pitfall 2: Trademark and packaging-image use treated as a copyright-only problem

**What goes wrong:**
The project necessarily names brands (Hovis, Heinz, McVitie's) and wants historic packaging images. Two distinct legal regimes apply and teams conflate them:
- **Trademark** (UK Trade Marks Act 1994): using a brand name to *refer to the brand's own products* is generally lawful (referential/honest use), but use that implies endorsement, denigrates the mark, or takes unfair advantage of its reputation is infringement. "Expectation vs Reality" framing that mocks a brand drifts from referential use toward denigration.
- **Copyright/design rights**: packaging artwork, logos and photographs are separately protected. A historic packshot scraped from the Internet Archive or a supermarket page is someone's copyrighted image — the Internet Archive hosting it does **not** license you to republish it.

The CC BY-SA on OFF photos covers OFF-contributed photos, **not** the brand artwork depicted in them — OFF's own terms warn that trademark, design and image rights on the packaging remain the reuser's responsibility.

**Why it happens:**
"It's on the public internet / in a public archive" is mistaken for "it's licensed for reuse". Brand names feel free to use because everyone uses them in reviews.

**How to avoid:**
- Establish an **image-rights policy per source** before ingesting any image: own photography, OFF CC BY-SA photos (with attribution + share-alike), or fair-dealing reliance (UK fair dealing is narrow — criticism/review/quotation/news reporting, with sufficient acknowledgement; it does **not** cover decorative use).
- Prefer **describing** historic formulations in text + structured data over republishing copyrighted packshots; where an image is essential, prefer own-photographed artefacts or explicitly-licensed images.
- Keep brand references **referential and neutral** (Key Decision already commits to "public archive, not campaigning" — enforce it in an editorial style guide so trademark denigration risk is designed out, not policed later).
- Maintain a per-asset rights ledger (source URL, licence, date captured, rights holder) alongside the trust layer.

**Warning signs:**
Bulk-downloaded packshots with no per-image licence field; UI copy that editorialises against a brand; "we'll sort out image rights later"; reliance on Internet Archive as if it were a licence.

**Phase to address:**
Foundation phase (rights ledger + editorial style guide) before any ingestion. Re-checked in the historical/timeline phase where old packaging imagery is most tempting.

### Pitfall 3: Scraping manufacturer and supermarket sites without a legal/technical sourcing policy

**What goes wrong:**
Ingredient and nutrition data is pulled by scraping manufacturer and supermarket sites. Risks: breach of site **Terms of Service** (which often prohibit scraping), copyright in the compiled content, database-right in supermarket catalogues (UK/EU **sui generis database right** protects substantial extraction), rate-limiting/IP bans, and silent breakage when sites change markup — which then corrupts the archive with stale or mis-parsed values.

**Why it happens:**
Scraping is the path of least resistance for breadth. The legal exposure (database right, ToS) is invisible until a cease-and-desist or a blocked IP arrives, and parser drift corrupts data quietly.

**How to avoid:**
- Prefer **licensed/official sources** in priority order: Open Food Facts (open), official manufacturer data feeds where offered, FSA/EFSA open datasets, then scraping only as a last resort and only where ToS permit.
- For any scraping: respect robots.txt and ToS, throttle, identify your agent, cache the **raw fetched artefact with a capture timestamp** so every parsed value is reproducible and auditable (essential for the trust layer's "evidence" claim).
- Build **parser-drift detection**: validate parsed values against expected ranges/schemas and alert on anomalies rather than writing them straight to the archive.
- Treat supermarket catalogue extraction as touching database right — extract facts (ingredients, nutrition per 100g are largely non-copyrightable facts), not the creative compilation.

**Warning signs:**
No stored raw source artefact (only the parsed value); scrapers with no rate-limiting or User-Agent; a single scraper feeding production with no validation layer; ingredient strings that look mis-parsed (truncated, HTML fragments).

**Phase to address:**
Data-ingestion/sourcing-pipeline phase. The raw-artefact-capture requirement should be a success criterion of that phase.

### Pitfall 4: Misrepresenting evidence and implying causation

**What goes wrong:**
Nutrition/additive content slides from "associated with" to "causes", reports a single study as settled science, cites EFSA/FSA positions that have since changed, or presents a contested area (e.g. some emulsifiers, sweeteners) as resolved in either direction. This is the fastest way to destroy the project's core value (trust) and invites both expert ridicule and regulatory scrutiny of health claims.

**Why it happens:**
Causal language is more readable than hedged language; the "transparency" brand creates pressure to give users a clear verdict; primary literature is hard, secondary summaries are tempting and often overstate.

**How to avoid:**
- Encode **evidence level as structured data** (e.g. regulatory position / systematic review / single study / mechanistic-only) and make the UI render the strength honestly — the trust layer already mandates an evidence level; enforce it as a non-nullable field with a controlled vocabulary.
- Anchor claims to **current regulatory positions** (FSA, EFSA health-claims register, GB additive authorisations) with a checked-on date, because positions and authorisations change (and GB and EU regimes have diverged post-Brexit — do not assume an EU authorisation is the GB position).
- A house rule: **no causal verb without a systematic-review-or-stronger source.** Default to "evidence suggests / is associated with / is inconclusive".
- Editorial review by the standard of "would a nutrition scientist sign this off?" before publish.

**Warning signs:**
"Causes/prevents/toxic" language with single-study citations; evidence-level field left blank or defaulted; citations to news articles rather than the underlying study; EFSA/FSA claims with no checked-on date; assuming EU = GB regulatory status.

**Phase to address:**
Evidence/methodology phase (controlled vocabulary for evidence levels + editorial review gate). The methodology page must publicly document how uncertainty is represented.

### Pitfall 5: Defamation and malicious-falsehood risk in "why a recipe changed" claims

**What goes wrong:**
Stating *why* a manufacturer reformulated ("they cut cocoa to boost margins", "they replaced butter with palm oil to cut costs") asserts motive. If unproven and damaging to the company's reputation, it risks **defamation** (UK Defamation Act 2013 — corporations must show serious financial harm, but the threshold is reachable for a public reference site) or **malicious falsehood**. Even true-but-unsourced motive claims are dangerous because **truth must be provable in court**, not merely believed.

**Why it happens:**
"Why it changed" is the most engaging part of the story and the hardest to source. Motive is rarely documented; the temptation is to infer it and state the inference as fact.

**How to avoid:**
- **Separate fact from inference in the data model and the UI.** A reformulation has: (a) the documented change (datable, sourceable), (b) the *stated* reason (quote the manufacturer/press release), and (c) any *analyst inference* — clearly labelled as inference/opinion, not fact.
- Lean on the **honest-opinion defence**: opinion clearly presented as opinion, based on stated facts, is defensible in a way that a bald factual assertion of motive is not.
- For sugar/salt reductions, the **honest-truth** anchor often exists (FSA reformulation programmes, sugar tax / Soft Drinks Industry Levy timelines) — cite the policy driver rather than imputing a cynical motive.
- When motive is genuinely unknown, **say so** ("the manufacturer has not stated why") — this is on-brand ("explain uncertainty honestly") and legally safest.

**Warning signs:**
Reformulation records with a "reason" field populated by inference but stored as fact; profit/greed motive claims with no citation; absence of a fact/opinion distinction in the schema.

**Phase to address:**
Historical/timeline-engine phase (schema must separate documented change, stated reason, and labelled inference). Editorial style guide from the foundation phase sets the rule.

### Pitfall 6: Historical gaps presented as certainty; false precision in dating reformulations

**What goes wrong:**
Old formulations are sourced from uneven evidence (one archived packshot, a forum memory, a single Wayback capture). The archive then states "reformulated in 2014" when the truth is "the recipe on the 2014-03 Wayback capture differs from the 2011-06 capture — the change happened *somewhere* in that window". Presenting a captured-date as a change-date is false precision; presenting an absence of evidence as evidence of absence ("contained no palm oil before 2015") is worse.

**Why it happens:**
Timelines demand specific dates to render. The Internet Archive gives a capture date, which is silently promoted to a "change date". Sparse history feels embarrassing, so gaps get smoothed over.

**How to avoid:**
- Model dates as **ranges with a basis**, not points: `change observed between [earliest evidence] and [latest evidence]`, plus the evidence type. The timeline UI must render uncertainty (fuzzy edges, "circa", confidence shading) rather than crisp false dates.
- Distinguish **"no evidence of X before date Y"** from **"X did not exist before Y"** in copy and schema — never let the UI imply the latter.
- Capture the **Wayback snapshot URL and capture timestamp** as the evidence, and make clear that a snapshot proves the recipe *as captured on that date*, not the date it changed.
- A "history completeness" indicator per product so sparse histories are honestly flagged, not hidden.

**Warning signs:**
Single point-dates on timeline entries with no uncertainty; "change date" equal to a Wayback capture date; absence-of-evidence phrased as fact; no completeness/coverage indicator on historical sections.

**Phase to address:**
Historical/timeline-engine phase (date-range modelling + uncertainty rendering is a success criterion). Trust-layer schema (foundation) must support ranged dates.

### Pitfall 7: The confidence model decays, gets gamed, or becomes decorative

**What goes wrong:**
"Every fact has a confidence level" is the differentiator, but confidence scores rot in predictable ways:
- **Anchoring inflation:** contributors default to "medium/high" because "low" looks bad, so the scale loses meaning.
- **No decay:** a "high confidence" 2019 nutrition value is still shown as high in 2026 though the product may have reformulated — confidence and freshness get conflated and neither is trustworthy.
- **Uncalibrated scale:** "confidence" means different things across contributors with no rubric, so it isn't comparable.
- **Theatre:** the badge is shown but nothing in the system enforces it (no rule blocks publishing a "high confidence" fact sourced from a forum post).

**Why it happens:**
Confidence is easy to display and hard to govern. Without a written rubric tying confidence to evidence type + source + recency, it becomes a vibe.

**How to avoid:**
- **Derive confidence from rules, not feelings.** Define confidence as a function of (source tier × evidence level × recency), documented on the methodology page and ideally computed, not hand-entered, so it is reproducible and calibrated.
- **Separate confidence from freshness.** Store `last_verified_date` distinctly and let the UI degrade displayed confidence (or flag "may be stale") as facts age past a category-specific threshold (nutrition panels change more often than, say, the founding date of a brand).
- Publish the **rubric** so the score is auditable — this is the whole trust proposition.
- Periodic **calibration audit**: sample facts, re-derive confidence, check the displayed badge still holds.

**Warning signs:**
Confidence entered as free choice with no rubric; distribution skewed to high (few "low"); no `last_verified` field; confidence unchanged on a 5-year-old record; methodology page describes confidence vaguely.

**Phase to address:**
Foundation phase (define the rubric and make confidence rule-derived; add `last_verified` + decay support). Methodology phase publishes the rubric.

### Pitfall 8: Staleness — a curated archive rots faster than it's maintained

**What goes wrong:**
Products reformulate, get discontinued, change owners; regulatory positions and additive authorisations change. A "100 iconic products" archive that was accurate at launch quietly becomes a museum of wrong current-state data, undermining trust. Curated depth (the project's strategy) is exactly what is most expensive to keep fresh.

**Why it happens:**
Initial curation is a burst of effort; maintenance is unglamorous and unfunded. There's no signal telling you which of 100 products went stale this quarter.

**How to avoid:**
- Make **freshness visible and queryable**: `last_verified` per fact, surfaced in the UI ("verified March 2026"), and an internal staleness dashboard sorted by age × volatility.
- **Tier re-verification by volatility**: nutrition/ingredients (high churn) on a short cadge; brand history/provenance (low churn) rarely. Don't try to re-verify everything equally.
- Automate **change detection** where possible (re-fetch OFF / source and diff) to flag candidates for human re-verification rather than relying on manual sweeps.
- Design the data model so a stale fact **degrades gracefully** (shows its age) rather than silently presenting as current — staleness honestly shown is on-brand; staleness hidden is a trust breach.
- Be honest in scope: deep histories for 100 products that are *maintained* beats 1,000 that rot.

**Warning signs:**
No `last_verified` surfaced to users; no staleness dashboard; "we'll refresh annually" with no mechanism; discontinued products still presented as current.

**Phase to address:**
Foundation phase (freshness as a first-class field). A dedicated maintenance/freshness phase or ongoing operational track before scaling product count.

### Pitfall 9: Citation-first content that search engines and assistive tech can't read

**What goes wrong:**
A reference site lives or dies on being **found and trusted**. Two failure modes:
- **SEO/E-E-A-T:** heavy citation content rendered client-side, no structured data, weak author/sourcing signals — Google can't establish the expertise/authority a YMYL ("Your Money or Your Life", which includes health/nutrition) topic is held to, and the site doesn't rank. Citations as raw text instead of marked-up references waste the trust signal.
- **Accessibility:** complex comparison tables, timelines and processing visualisations built without semantics fail WCAG 2.2 AA — screen-reader users can't parse the comparison, keyboard users can't operate the timeline, charts have no text alternative. For a public-interest reference site this is both an exclusion and (for some operators) a legal exposure.

**Why it happens:**
SEO and accessibility are treated as a final polish. Data-dense UI (tables, timelines, multi-dimensional processing displays) is the hardest to make accessible and is exactly the project's core UI.

**How to avoid:**
- **Server-render** the citation content; emit structured data (Schema.org `Article`/`Dataset`/`ClaimReview`-style and `Citation` markup) so sources are machine-readable and E-E-A-T signals (author, sourcing, dates) are explicit.
- Treat the project as **YMYL**: clear authorship, methodology page, dated sources, and named editorial standards are ranking *and* trust requirements, not nice-to-haves.
- Build data visualisations **semantic-first**: real `<table>` markup with headers for comparisons; timelines and processing displays with text-equivalent representations, keyboard operability, visible focus, and 4.5:1 contrast (per the project's WCAG 2.2 AA default). Verify with axe-core/pa11y, don't assume.
- Multi-dimensional processing display (a deliberate Key Decision) is the highest accessibility risk — design its non-visual representation up front.

**Warning signs:**
Client-only rendering of reference content; citations as plain text with no markup; comparison data in CSS-styled `<div>`s not tables; timeline/processing widgets that can't be tab-navigated; no automated accessibility check in the pipeline.

**Phase to address:**
Publishing/front-end phase (SSR + structured data + accessibility as success criteria, verified with automated tooling). Processing-explorer and timeline phases each own the accessibility of their own widget.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Store OFF-derived values in the same fields as editorial values, no source discriminator | Faster ingest, simpler schema | ODbL contamination invisible until API/export; rights ledger impossible to reconstruct | Never — provenance tagging is the core value |
| Hand-entered confidence with no rubric | Ship the badge fast | Scale becomes meaningless, uncalibrated, ungovernable | Never for a trust-first product |
| Store parsed value only, discard raw fetched source | Less storage | Claims unauditable; parser bugs uncorrectable; trust claim hollow | Never — raw artefact is the evidence |
| Point-dates on historical timeline (use Wayback capture date as change date) | Timeline renders cleanly | False precision; corrects into a credibility scandal | Never — model ranges from day one |
| Republish scraped packshots, "sort rights later" | Visual richness fast | Copyright/trademark exposure across whole image library | Never |
| Client-side-only rendering of reference content | Simpler SPA build | YMYL ranking failure; assistive-tech exclusion | Only for non-content admin UI |
| Single overall processing score (explicitly rejected) | Easy to display/sort | Oversimplifies, misleads, contradicts methodology | Never (already a Key Decision) |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Open Food Facts (ODbL) | Attribution-only assumption; commingling with editorial data; serving raw OFF DB via API | Tag source per field; render as Produced Work with ODbL notice; gate any DB distribution behind a licence review |
| OFF product photos (CC BY-SA 3.0) | Treating as freely reusable; ignoring share-alike; assuming it licenses the brand artwork in the photo | Attribute + share-alike the photo; verify packaging trademark/design rights separately |
| Internet Archive / Wayback | Treating hosting as a licence; capture date = change date | Cite snapshot URL + timestamp as evidence of state-on-date only; verify underlying image rights |
| Manufacturer / supermarket sites | Scraping against ToS; ignoring DB right; no parser-drift detection | Prefer official feeds; respect ToS/robots; extract facts not compilations; store raw artefact; validate ranges |
| EFSA / FSA positions | Assuming EU authorisation = GB position (post-Brexit divergence); no checked-on date | Use GB regulatory source for GB claims; stamp every regulatory fact with checked-on date |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Comparison/search querying across fully-joined fact+source+confidence tables | Slow product/ingredient comparison as fact count grows | Denormalise read models for display; index source/confidence; precompute comparisons | Hundreds of products × many facts each |
| Re-deriving confidence/freshness on every page render | Slow pages, inconsistent badges | Compute and cache derived confidence; recompute on fact update or scheduled decay sweep | Once decay logic is non-trivial |
| Full-archive re-verification sweeps | Maintenance becomes unaffordable, stops happening | Volatility-tiered cadence + change-detection diffs flag only candidates | As product count grows past manual capacity |
| Unindexed full-text search across ingredients/evidence prose | Slow, irrelevant search results | Proper search index with fielded boosting (product/ingredient/brand) | As corpus grows beyond trivial |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Scraper credentials / API keys in repo or client | Source access revoked; abuse | Server-side secrets; never ship keys to the browser |
| No rate-limit / identification on scrapers | IP bans, legal complaints, source loss | Throttle, identify agent, respect robots/ToS, cache to avoid re-fetching |
| Trusting scraped/OFF strings as safe HTML | Stored XSS via ingredient/source fields rendered unescaped | Treat all external data as untrusted; escape on output; sanitise |
| Public-facing data export with no licence/PII review | Inadvertent ODbL breach; leaking contributor data (Phase 2) | Licence + privacy review gate before any export/API release |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Confidence/evidence badges shown without explanation | Users can't interpret them; badge becomes noise | Link every badge to the methodology rubric; plain-language tooltips |
| Crisp dates on uncertain history | Users over-trust precision that isn't there | Render uncertainty (circa, ranges, shading); show history completeness |
| Outrage/"shocking truth" framing | Contradicts neutrality; erodes trust with experts/journalists | Neutral, comparative, sourced presentation (already a Key Decision) |
| Multi-dimensional processing shown as if a verdict | Users want a score and misread the dimensions as one | Explain each dimension; explicitly state why no single score exists |
| Stale data with no visible date | Users act on out-of-date nutrition/ingredients | Surface `last_verified` everywhere; flag/age-degrade stale facts |
| Citations hidden behind clicks | Defeats the citation-first promise | Sources visible and linked inline at point of claim |

## "Looks Done But Isn't" Checklist

- [ ] **Trust layer:** Every fact actually carries source + confidence + evidence level + last_verified — verify there are no nullable/defaulted trust fields in production data, not just the schema.
- [ ] **OFF attribution:** Required ODbL notice + per-product source links present on rendered pages — verify wording and links, not just "we credit OFF somewhere".
- [ ] **Image rights:** Every published image has a source + licence + rights-holder record — verify the ledger covers 100% of live images.
- [ ] **Raw source artefacts:** Each scraped/ingested fact has a stored, timestamped raw source — verify you can reproduce any displayed value from its evidence.
- [ ] **Historical dates:** Timeline entries store ranges + evidence basis, not bare points — verify no change-date silently equals a capture date.
- [ ] **Fact vs inference:** "Why it changed" content separates documented change, stated reason, and labelled inference — verify no inferred motive is stored/shown as fact.
- [ ] **Regulatory currency:** Additive/health claims cite GB (not assumed-EU) positions with checked-on dates — verify divergence cases.
- [ ] **Accessibility:** Comparison tables, timeline, processing explorer pass axe-core/pa11y and keyboard navigation — verify the data-dense widgets specifically, not just static pages.
- [ ] **SEO/E-E-A-T:** Reference content server-rendered with structured data, authorship and methodology signals — verify with a crawl/rich-results test.
- [ ] **Methodology page:** Publicly documents the confidence rubric and how uncertainty is represented — verify it matches what the system actually does.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| ODbL contamination discovered at API time | HIGH | Audit which fields are OFF-derived; either segregate and export only own-data, release the derivative DB under ODbL, or restrict the API to Produced-Work outputs |
| Commingled OFF/editorial data, no source tags | HIGH | Re-derive provenance from raw artefacts if retained; otherwise re-source — a strong argument for tagging from day one |
| Trademark/copyright complaint on an image | MEDIUM | Take down promptly; replace with own/licensed asset or text description; audit the rest of the library against the same source |
| Defamation/falsehood complaint on a "why changed" claim | MEDIUM–HIGH | Reframe as labelled opinion grounded in stated facts, or retract if unprovable; correct and date the correction publicly |
| Confidence scores found uncalibrated | MEDIUM | Define rubric retroactively; batch re-derive from source tier × evidence × recency; recompute badges |
| Archive found widely stale | MEDIUM | Run volatility-tiered re-verification; surface `last_verified`; flag unverified facts rather than hiding the problem |
| False-precision historical dates | LOW–MEDIUM | Migrate point-dates to ranges from retained evidence; update timeline rendering to show uncertainty |
